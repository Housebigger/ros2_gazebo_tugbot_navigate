# Phase123 maze_explorer first-goal dispatch design review

Status: DESIGN_ONLY - Phase123 doc-only/design-only review. Phase124 not entered.

## Purpose

Phase122 completed a post-ingress handoff dry-start. The Phase120 same-run ingress chain SUCCEEDED, the Phase122 handoff gate passed, and `maze_explorer` was started only with `max_goals=0`. That dry-start proved the post-ingress handoff boundary can start `maze_explorer` safely without dispatching exploration goals.

Phase122 evidence was:

```text
Phase120 ingress succeeded
Phase122 handoff_allowed=true
Nav2 idle
scan/TF/costmap fresh
ingress_goal_result.result_status_label=SUCCEEDED
maze_explorer_started=true
maze_explorer_max_goals=0
exploration_goal_dispatched=false
goal_event_count=0
dispatch_event_count=0
goal_active=false
```

Phase123 defines the design contract for future Phase124: from the same handoff-ready boundary, allow `maze_explorer` to run with `max_goals=1` and dispatch the first exploration goal only. Phase123 does not implement that runtime and does not start any ROS/Gazebo process.

A future Phase124 success is first-goal smoke success only. It does not represent autonomous exploration success, does not represent exit success, does not validate Goal1 carry-over, does not validate staging success, and does not validate full-maze exploration.

## Inputs from Phase120/Phase122

Future Phase124 must treat Phase120 and Phase122 as required same-run prerequisites, not as stale historical facts.

Required preconditions before allowing first-goal dispatch:

1. Phase120 ingress succeeded in the same visible stack run.
   - same-run readiness wait passed;
   - strict preflight passed;
   - the only explicit ingress goal was `frame_id=map`, `x=2.0`, `y=0.0`, `yaw=0.0`;
   - ingress goal was accepted;
   - `ingress_goal_result.result_status_label=SUCCEEDED`.
2. Phase122 handoff-ready evidence is re-collected or carried forward from the same runner transaction.
   - `Phase122 handoff_allowed=true`;
   - robot pose is near the ingress target;
   - `Nav2 idle` is true with no active/pending/executing/canceling NavigateToPose goal;
   - `scan/TF/costmap fresh` is true;
   - costmap samples use the Phase122 late-subscriber QoS lesson where needed.
3. Phase122 dry-start invariant is known before Phase124 changes the goal budget.
   - `maze_explorer_started=true` was previously demonstrated only for `maze_explorer_max_goals=0`;
   - `exploration_goal_dispatched=false`;
   - `goal_event_count=0`;
   - `dispatch_event_count=0`;
   - `goal_active=false`.

If any required handoff evidence is missing, stale, contradictory, out of tolerance, or not same-run, Phase124 must classify `HANDOFF_NOT_READY_NO_EXPLORATION`, must not start `maze_explorer` with a dispatch-capable budget, and must not send any exploration goal.

## Future Phase124 allowed scope

Future Phase124 may do exactly this and no more:

- launch visible Gazebo/RViz/SLAM/Nav2;
- reproduce the Phase120 managed-active readiness wait, strict preflight, and explicit inner-ingress goal;
- verify the Phase122 handoff gate is still ready;
- start `maze_explorer` with `max_goals=1` only after handoff is ready;
- allow one first exploration goal only;
- observe the first dispatch, acceptance/result/abort/timeout, and diagnostic evidence;
- stop immediately after first terminal outcome.

Future Phase124 must not keep exploring after that first goal. The Phase124 stop condition is reached when one of these happens:

- first exploration goal is dispatched and accepted by Nav2;
- first exploration goal is rejected by Nav2;
- first exploration goal times out;
- no valid first goal can be generated inside the bounded observation window;
- handoff is not ready, so no exploration is allowed.

## Source of the first goal

The first goal must be generated by the current maze_explorer algorithm. It is not a hand-authored pose, not a hand-selected Goal1, and not an externally supplied target.

Required source rules:

- use `maze_explorer` current branch/topology/frontier logic as-is;
- do not manually specify Goal1;
- do not inject a fixed first-goal coordinate;
- do not use fallback goal as the first-goal validation target; this is not fallback goal validation;
- do not use terminal goal as the first-goal validation target; this is not terminal goal validation;
- do not use exit goal as the first-goal validation target; this is not exit goal validation;
- do not use near-exit fallback/micro-goal evidence as first-goal success;
- the dispatched first-goal event must be `goal_kind=explore`;
- branch scoring, centerline, fallback, and terminal acceptance behavior must remain unchanged.

If the first dispatch is not `goal_kind=explore`, Phase124 must classify it as a diagnostic failure, not as first exploration goal success.

## Classification contract

Future Phase124 must use exactly these high-level classifications:

```text
FIRST_EXPLORATION_GOAL_DISPATCHED_ACCEPTED_STOP
FIRST_EXPLORATION_GOAL_REJECTED_DIAGNOSTIC_FAIL
FIRST_EXPLORATION_GOAL_TIMEOUT_DIAGNOSTIC_FAIL
NO_VALID_FIRST_GOAL_FAIL_CLOSED
HANDOFF_NOT_READY_NO_EXPLORATION
```

Meanings:

- `FIRST_EXPLORATION_GOAL_DISPATCHED_ACCEPTED_STOP`: handoff was ready; `maze_explorer` ran with `max_goals=1`; exactly one first `goal_kind=explore` dispatch was observed; Nav2 accepted the goal; the runner stopped at the bounded first-goal smoke boundary. This is first-goal smoke success only.
- `FIRST_EXPLORATION_GOAL_REJECTED_DIAGNOSTIC_FAIL`: handoff was ready and a first `goal_kind=explore` dispatch attempt happened, but Nav2 rejected the goal or no accepted goal handle was obtained. Preserve candidate, pose, selection reason, frontier/topology evidence, and rejection diagnostics.
- `FIRST_EXPLORATION_GOAL_TIMEOUT_DIAGNOSTIC_FAIL`: handoff was ready and the first `goal_kind=explore` dispatch was accepted, but the bounded result window timed out or `maze_explorer` emitted a timeout outcome. Preserve Nav2 feedback, progress, timeout local-cost evidence, scan/map/odom/TF, and candidate diagnostics.
- `NO_VALID_FIRST_GOAL_FAIL_CLOSED`: handoff was ready but no valid first `goal_kind=explore` candidate/dispatch emerged during the bounded observation window. Preserve local topology, frontier/candidate counts, readiness gate, map/local-cost, and no-candidate reasons.
- `HANDOFF_NOT_READY_NO_EXPLORATION`: Phase120/Phase122 preconditions were not satisfied; no dispatch-capable `maze_explorer` startup is allowed and no exploration goal may be sent.

All non-success outcomes are fail-closed. Do not convert timeout/failure into success; in analyzer terms, do not convert timeout/failure into success. A rejected, aborted, canceled, stale, fallback, terminal, exit, or non-`explore` goal does not satisfy `FIRST_EXPLORATION_GOAL_DISPATCHED_ACCEPTED_STOP`.

## Guardrails for the future Phase124 runner

Future Phase124 runner/analyzer must enforce:

```text
max_goals=1
first exploration goal only
handoff_ready_required=true
only_current_algorithm_goal_source=true
manual_goal1_forbidden=true
fallback_goal_forbidden_as_success=true
terminal_goal_forbidden_as_success=true
exit_goal_forbidden_as_success=true
branch_scoring_changed=false
centerline_gate_changed=false
fallback_changed=false
terminal_acceptance_changed=false
nav2_config_changed=false
```

A future runner may observe `maze_explorer` goal events and state topics, but it must stop at the first terminal outcome for the first exploration goal. It must not continue autonomous exploration after the first goal, even if that first goal succeeds.

## Analyzer design for future Phase124

Future Phase124 analyzer should validate:

1. Handoff readiness:
   - Phase120 ingress succeeded;
   - Phase122-style `handoff_allowed=true`;
   - robot pose near ingress target;
   - Nav2 idle;
   - scan/TF/costmap fresh.
2. Startup:
   - `maze_explorer` started only after handoff ready;
   - `max_goals=1`;
   - dispatch-capable startup did not happen if handoff was not ready.
3. First-goal source:
   - first dispatch comes from `/maze/goal_events` or equivalent `maze_explorer` diagnostics;
   - `goal_kind=explore`;
   - generated by the current maze_explorer algorithm;
   - not manually specified Goal1;
   - not a fallback/terminal/exit goal.
4. Evidence completeness:
   - candidate, pose, frame_id, selection reason, frontier evidence, topology evidence;
   - dispatch time;
   - accepted/rejected/timeout/result fields;
   - Nav2 feedback;
   - local_costmap samples;
   - scan/map/odom/TF.
5. Success boundary:
   - success means first-goal smoke success only;
   - no autonomous exploration success claim;
   - no exit success claim.
