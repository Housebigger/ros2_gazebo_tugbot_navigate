## High-Level Strategy

### Strategy A: Keep 0514 exploration chain as baseline

先让 0522 在迷宫 world 中跑通 0514 的链路：

```text
Gazebo maze world
  -> /scan /odom /tf /cmd_vel /clock bridge
  -> slam_toolbox live /map
  -> Nav2 no-spin params + BT
  -> frontier/perimeter explorer sends NavigateToPose
  -> map grows while robot explores maze
```

这一步目标不是立刻完美出迷宫，而是证明继承链路没有损坏。

### Strategy B: Add maze-specific completion criterion

0514 的探索完成条件偏向“建图完成 / unknown ratio 降低”。0522 的核心完成条件应改为：

```text
robot reaches exit region and exits maze boundary
```

因此需要新增 `maze_goal_monitor`：

- 输入：TF `map -> base_link` 或 Gazebo truth pose。
- 配置：出口线段 / 出口区域 polygon / 出口外侧区域。
- 输出：日志、topic、可选 lifecycle shutdown signal。
- 验收：机器人越过出口线或进入出口区域，记录 `MAZE_EXIT_REACHED`。

### Strategy C: Add maze-specific navigation fallback

普通 frontier 在迷宫中可能遇到：

- 死胡同多，goal 频繁失败。
- 局部代价地图过窄导致目标不可达。
- 只追 frontier 可能来回震荡。
- no-spin 虽保护地图，但也降低局部观察完整性。

0522 推荐采用“保守继承 + 小步增强”：

1. 第一阶段只复用 `frontier_explorer`，加 maze world 和 exit monitor。
2. 如果不能稳定出迷宫，再新增 `maze_explorer.py`，实现 frontier + graph memory + dead-end avoidance。
3. 最后再考虑 wall-follow fallback，不要一开始推翻 Nav2 goal-based 结构。

Recommended maze explorer behavior:

```text
live /map + robot pose
  -> identify corridor frontier / junction candidates
  -> maintain visited goal graph and failed/dead-end blacklist
  -> prefer unexplored corridor frontiers aligned with reachable free space
  -> avoid repeatedly entering known dead ends
  -> if exit monitor detects success, stop exploration and save final map
```

### Phase 4: Convert Maze Image into Gazebo World

**Objective:** 从迷宫图片生成可仿真的 Gazebo maze world。

**Files:**
- Create: `src/tugbot_maze/tugbot_maze/maze_image_to_world.py`
- Modify/Create: `src/tugbot_maze/config/maze_instance.yaml`
- Generate/Create: `src/tugbot_gazebo/worlds/tugbot_maze_world.sdf`
- Possibly copy: `src/tugbot_maze/assets/maze_instance.png`

**Image-to-world design:**

Input config should include:

```yaml
image: package://tugbot_maze/assets/maze_instance.png
resolution_m_per_pixel: 0.05
wall_height_m: 1.2
wall_thickness_min_m: 0.05
free_threshold: 200
occupied_threshold: 80
start:
  method: manual
  x_m: 0.0
  y_m: 0.0
  yaw_rad: 0.0
exit:
  method: manual
  x_m: 5.0
  y_m: 0.0
  radius_m: 0.5
```

World generation approach:

1. Load image.
2. Convert to binary occupancy:
   - dark pixels -> wall
   - light pixels -> free
3. Optionally downsample or merge wall pixels into rectangular blocks to avoid huge SDF files.
4. Generate static wall boxes with collision + visual.
5. Generate floor plane.
6. Spawn Tugbot at entrance pose.
7. Add optional exit visual marker that does not affect navigation.
8. Preserve world-level sensors system and physics setup from 0514 `tugbot_nav_world.sdf`.

**Important tradeoff:**

- Pixel-per-wall box is easy but may create too many SDF links.
- Rectangular merging is better. Use row-run / connected rectangle merging before writing SDF.
- Start with simple but bounded merging. If maze image is small, pixel boxes may be acceptable for first smoke test.

**Verification:**

- `tugbot_maze_world.sdf` exists.
- SDF contains Tugbot include/model and maze wall boxes.
- Gazebo can launch without crashing.
- `/scan` sees nearby maze walls.

**Objective:** 迷宫比 0514 world 更窄，可能需要专用 Nav2 参数。

**Candidate tuning areas:**

1. footprint / robot_radius:
   - Ensure costmap radius matches Tugbot footprint.
   - Do not unrealistically shrink robot just to pass walls.

2. inflation_radius:
   - Too large blocks narrow corridors.
   - Too small risks wall contact.
   - Start with conservative values and document.

3. resolution:
   - Maze world and SLAM map should have compatible scale.

4. controller:
   - Keep low angular velocity to protect SLAM.
   - But maze turns may need enough yaw authority.

5. goal tolerance:
   - Junction/corridor goals may need looser yaw tolerance.

**Verification:**

- Manual short goals through one turn succeed.
- Robot does not collide with walls.
- Costmaps do not fully close passable corridors.

### Phase 9: Add Maze-Specific Explorer Only If Needed

**Objective:** 如果 inherited frontier/perimeter 不能稳定出迷宫，则新增 `maze_explorer.py`，但不要替换 0514 explorer。

**Files:**
- Create: `src/tugbot_maze/tugbot_maze/maze_explorer.py`

- Modify: `src/tugbot_bringup/launch/tugbot_maze_explore.launch.py`
- Test: `src/tugbot_maze/test/test_maze_explorer_contract.py`

**Recommended minimal maze algorithm:**

1. Subscribe to `/map` OccupancyGrid.
2. Get robot pose via TF.
3. Extract free-space skeleton or corridor frontiers from live map.
4. Build candidate goals from:
   - frontier cells at corridor ends
   - unexplored junction directions
   - safe free cells near unknown boundary
5. Maintain memory:
   - visited goal positions
   - failed goal blacklist
   - dead-end zones
   - last progress timestamp
6. Score candidates:
   - prefer unexplored frontier
   - prefer forward/corridor continuation
   - penalize revisits
   - penalize goals near known dead ends
   - ensure clearance from occupied cells
7. Send `NavigateToPose` goals.
8. Stop immediately if `/maze/exit_reached` true.

**Avoid overengineering in first version:**

- Do not implement full graph SLAM.
- Do not compute global path from original maze image.
- Do not replace Nav2 low-level controller.
- Do not use direct `/cmd_vel` wall following unless Nav2 goal-based strategy proves insufficient.

**Verification:**

- Contract tests verify key parameters/logs/actions exist.
- Headless smoke sends at least one maze goal.
- Visual run reaches deeper maze regions than inherited frontier if baseline failed.

## Risks / Tradeoffs

1. `tmp_resources/` image currently not visible to tools.
   - Risk: implementation cannot start from actual maze asset.
   - Mitigation: first implementation step must re-check files; if still absent, ask user for filename or image placement.

2. Image-to-world scale mismatch.
   - Risk: corridors too narrow for Tugbot footprint or too large for efficient exploration.
   - Mitigation: choose explicit `resolution_m_per_pixel`; validate corridor width against robot radius + inflation radius.

3. Frontier exploration may not solve maze exit reliably.
   - Risk: robot explores many dead ends or loops.
   - Mitigation: start with inherited explorer for baseline, then add maze-specific memory/dead-end avoidance only if needed.

4. Nav2 costmap inflation may close maze corridors.
   - Risk: planner sees no valid path in narrow passages.
   - Mitigation: create `nav2_maze_slam_params.yaml` with maze-specific inflation/footprint tuning.

5. no-spin reduces observation in tight corners.
   - Risk: robot may miss side passages.
   - Mitigation: keep no-spin default, but allow small controlled yaw sampling if needed; never restore large 360-degree spins by default.

6. Directly using maze image for planning would violate autonomous exploration intent.
   - Risk: solution becomes offline path following rather than exploration.
   - Mitigation: image can define world and validation truth, not runtime path.

7. Gazebo SDF size could become too large if every wall pixel becomes a box.
   - Risk: slow launch or crash.
   - Mitigation: merge wall pixels into rectangles or downsample image.
